# 13_推理与服务脚本（加载/生成/流式/工具解析/模型转换）

本章覆盖推理与服务侧脚本（不训练，只加载权重、构造 prompt、生成、解析 `<think>`/`<tool_call>`、以及模型格式转换）。

涉及文件：

- `minimind_src/eval_llm.py`
- `minimind_src/scripts/serve_openai_api.py`
- `minimind_src/scripts/chat_api.py`
- `minimind_src/scripts/web_demo.py`
- `minimind_src/scripts/eval_toolcall.py`
- `minimind_src/scripts/convert_model.py`

这些脚本共同遵循一个核心约定：

- 使用 `transformers` tokenizer 的 `apply_chat_template(...)` 把多轮对话打平成“模型可读的文本模板”
- 再调用 `tokenizer(..., return_tensors="pt")` 得到 `input_ids`/`attention_mask`
- 通过 `model.generate(...)` 做自回归生成
- 对生成输出进行字符串层面的解析：提取 `<think>...</think>`（推理内容）与 `<tool_call>...</tool_call>`（工具调用 JSON）

## 1. 本地推理：`eval_llm.py`

### 1.1 `init_model(args)`：选择权重格式并加载

关键代码：

```python
12: def init_model(args):
13:     tokenizer = AutoTokenizer.from_pretrained(args.load_from)
14:     if 'model' in args.load_from:
15:         model = MiniMindForCausalLM(MiniMindConfig(...))
21:         ckp = f'./{args.save_dir}/{args.weight}_{args.hidden_size}{moe_suffix}.pth'
23:         model.load_state_dict(torch.load(ckp, map_location=args.device), strict=True)
24:         if args.lora_weight != 'None':
25:             apply_lora(model)
26:             load_lora(model, f'./{args.save_dir}/{args.lora_weight}_{args.hidden_size}.pth')
28:     else:
29:         model = AutoModelForCausalLM.from_pretrained(args.load_from, trust_remote_code=True)
30:     return model.half().eval().to(args.device), tokenizer
```

维度/状态说明：

- `tokenizer` 负责把文本模板切成 token
- 如果走原生 torch 权重路径（`'model' in args.load_from`）：
  - 构造 `MiniMindForCausalLM(MiniMindConfig(...))`
  - `load_state_dict` 把权重写入模型参数
  - 若启用 LoRA：先 `apply_lora(model)` 注入 LoRA 子模块，再 `load_lora(...)` 只加载 LoRA 的参数（底座权重仍在）
- 返回的 `model` 在生成时会产生：
  - `input_ids: [1, T]`（这里 eval 脚本按单样本推理）
  - `generated_ids: [1, T+new]`

### 1.2 `main()`：对话模板 -> 生成 -> decode

```python
71: conversation.append({"role":"user","content":prompt})
73: if 'pretrain' in args.weight:
74:     inputs = tokenizer.bos_token + prompt
76:     inputs = tokenizer.apply_chat_template(..., add_generation_prompt=True, open_thinking=...)
78: inputs = tokenizer(inputs, return_tensors="pt", truncation=True).to(args.device)

84:     max_new_tokens=args.max_new_tokens,

86:     pad_token_id=tokenizer.pad_token_id, eos_token_id=tokenizer.eos_token_id,

88: )

89: conversation.append({"role":"assistant","content":response})
```

维度追踪（eval 时最常见）：

- `inputs["input_ids"]`：`[1, T]`
- `generated_ids`：`[1, T + R]`
- `generated_ids[0][len(inputs["input_ids"][0]):]`：取后缀 `[R]` 进行 decode，得到最终回答字符串

## 2. OpenAI 兼容服务：`serve_openai_api.py`

该脚本用 `FastAPI` 提供 `/v1/chat/completions`，并支持 `stream=true` 时使用 Server-Sent Events（SSE）逐块输出。

### 2.1 请求模型 `ChatRequest`

```python
50: class ChatRequest(BaseModel):
51:     model: str
52:     messages: list
...
56:     stream: bool = True
57:     tools: list = []
58:     open_thinking: bool = False
59:     chat_template_kwargs: dict = None
61:     def get_open_thinking(self) -> bool:
62:         if self.open_thinking: return True
63:         if self.chat_template_kwargs:
64:             return self.chat_template_kwargs.get('open_thinking', False) or ...
68:         return False
```

要点：

- `tools`：直接透传给 tokenizer 的 `apply_chat_template(..., tools=...)`
- thinking 开关同时兼容 `open_thinking` 与 `chat_template_kwargs` 两种参数来源

### 2.2 字符流解析：`CustomStreamer.on_finalized_text`

它把 tokenizer 产生的增量文本塞进一个 `Queue`，用于异步生成循环：

```python
71: class CustomStreamer(TextStreamer):
77:     def on_finalized_text(self, text, stream_end=False):
78:         self.queue.put(text)
79:         if stream_end:
80:             self.queue.put(None)
```

维度说明：该脚本不在张量层面处理增量，只是字符串流（`text: str`）。

### 2.3 解析生成内容：`parse_response(text)`

这个函数把模型输出从“原始字符串”拆成：

- `content`: 去掉 `<think>` 和 `<tool_call>...</tool_call>` 后的最终可见答案文本
- `reasoning_content`: `<think>...</think>` 中的推理片段（若存在）
- `tool_calls`: 一组工具调用结构（OpenAI function-calling 兼容格式）

关键代码：

```python
84: def parse_response(text):
85:     think_match = re.search(r'<think>(.*?)</think>', text, re.DOTALL)
86:     if think_match:
87:         reasoning_content = think_match.group(1).strip()
88:         text = re.sub(r'<think>.*?</think>\s*', '', text, flags=re.DOTALL)
...

96:         call = json.loads(m.strip())

98:             "id": f"call_{int(time.time())}_{i}",
99:             "type": "function",
100:             "function": {"name": call.get("name",""), "arguments": json.dumps(call.get("arguments",{}), ensure_ascii=False)}
101:         })

102:         text = re.sub(r'<tool_call>.*?</tool_call>', '', text, flags=re.DOTALL)

```

### 2.4 生成流：`generate_stream_response(...)`

核心链路（字符串维度为主）：

```python

117:             do_sample=True,

120:             attention_mask=inputs.attention_mask,
121:             pad_token_id=tokenizer.pad_token_id,
122:             eos_token_id=tokenizer.eos_token_id,
123:             streamer=streamer
124:         )

128:     full_text = ""

133:         text = queue.get()
134:         if text is None: break

...

140:             if pos >= 0:
```

- `inputs.input_ids`: `[1, T]`
- streaming 每次取到 `text: str`，并在 yield 时拼成 OpenAI 兼容 delta 块

### 2.5 非流式分支

当 `request.stream=False`，它一次性 `model.generate(...)` 到 `generated_ids`，再：

```python

```

再组装 OpenAI 风格的 `choices[0].message` 返回。

## 3. 本地 OpenAI 客户端示例：`chat_api.py`

这是一个纯客户端脚本（并不关心张量）：

- 使用 `OpenAI(base_url="http://localhost:11434/v1")`
- 每轮把 `conversation_history` 追加 `{"role":"user" ...}`
- 调用 `client.chat.completions.create(..., stream=True, extra_body={...})`

流式部分核心是：

- `chunk.choices[0].delta` 中可能包含：
  - `reasoning_content`（对应服务端 `delta` 的 `reasoning_content`）
  - `content`（正文增量）

并最终把 assistant 完整内容写回 `conversation_history`。

## 4. ToolCall 交互评测：`eval_toolcall.py`

该脚本用于验证 `<tool_call>...</tool_call>` 是否能被正确解析、执行，并把工具结果继续喂给模型。

关键点集中在三个函数：

### 4.1 `parse_tool_calls(text)`

用正则抽取 `<tool_call> ... </tool_call>` 内的 JSON：

```python
70: def parse_tool_calls(text):
71:     matches = re.findall(r'<tool_call>(.*?)</tool_call>', text, re.DOTALL)
73:     for m in matches:
74:         calls.append(json.loads(m.strip()))
78:     return calls
```

### 4.2 `execute_tool(call, arguments=None)`

- 从 call 中取 `name` 与 `arguments`
- 在 `MOCK_RESULTS` 里按 name 找到对应函数并执行
- 失败时返回 `{"error": "..."}`

### 4.3 `run_case(...)`：多轮“工具回填”直到无 tool_call

主循环：

```python
179: while True:
180:     if args.backend == 'local':
181:         content = generate(...)
182:         tool_calls = parse_tool_calls(content)
...
185:     if not tool_calls: break
187:     tool_calls = [{"id":..., "name":..., "arguments":...}, ...]
192:     messages.append({"role":"assistant","content":content, ...})
193:     for tc in tool_calls:
196:         result = execute_tool(...)
199:         messages.append({"role":"tool","content": json.dumps(result, ensure_ascii=False), ...})
```

这里最重要的是 message role 约定：

- `{"role":"assistant","content":...}`：模型输出（可能含 tool_call 标签）
- `{"role":"tool","content": "...json..."}`：工具执行结果（作为“下一轮上下文的一部分”）

## 5. Streamlit Web Demo：`web_demo.py`

这是交互式网页端 Demo，重点在：

1. 对生成文本进行 UI 格式化（折叠显示思考、渲染 tool_call 标签）
2. 流式生成时实时刷新答案
3. 若检测到工具调用标签，则最多循环 16 次：解析 tool_call -> 执行工具 -> 回填 tool 结果 -> 重新生成

### 5.1 `process_assistant_content(content, is_streaming=False)`

它主要做三类渲染：

- 把 `<tool_call>{...}</tool_call>` 替换为带样式的 HTML 卡片
- 若启用 thinking：
  - 在 `</think>` 出现前，把思考内容放入 `<details open>...</details>` 并折叠展示
- 若 `<think>` 已完整出现，则渲染为“已思考”折叠块

### 5.2 流式生成与 token 维度

生成部分：

```python
360: inputs = tokenizer(new_prompt, return_tensors="pt", truncation=True).to(device)
362: streamer = TextIteratorStreamer(tokenizer, skip_prompt=True, skip_special_tokens=True)
364: generation_kwargs = {
365:     "input_ids": inputs.input_ids,  # [1, T]
366:     "max_length": inputs.input_ids.shape[1] + st.session_state.max_new_tokens,
367:     ...
373:     "streamer": streamer,
374: }
376: Thread(target=model.generate, kwargs=generation_kwargs).start()
379: for new_text in streamer: ...
```

同样，这里增量是 `new_text: str`，不是 tensor。

### 5.3 工具回填重生成

当答案中出现 `<tool_call>...</tool_call>`：

- 用正则 `re.findall(...)` 抽取多个 tool_call
- 对每个 tool_call：`json.loads` -> `execute_tool(name,args)` -> 把 tool result 写进 `full_answer` 并调用 `placeholder.markdown(...)`
- 然后重新构造 prompt：
  - `tokenizer.apply_chat_template(st.session_state.chat_messages, **template_kwargs)`
  - 再 `model.generate(...)` 继续补全

## 7. 本章小结：推理侧的“字符串协议”

在整个推理/服务侧，需要特别记住三种“文本协议标签”的含义：

- `<think>...</think>`：推理内容（服务端可拆成 `reasoning_content`，Web 端可折叠显示）
- `<tool_call>{json}</tool_call>`：工具调用（服务端/评测脚本解析后执行工具）
- tool result 以 `{"role":"tool","content": "json字符串"}` 的形式回填到 messages

这些标签让训练阶段的 RL（对 agent 的多轮交互/奖励）与推理阶段的 tool_call 执行链路保持一致。
