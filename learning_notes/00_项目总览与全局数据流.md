## 1. minimind 到底在做什么（极简原理）

`minimind` 是一个以“自回归语言建模 + 多种对齐训练范式（SFT / DPO / 蒸馏 / RL）+ 可选 MoE 与 LoRA 适配”为主线的项目。

整体可以抽象成四条主链路：

1. **训练链路**（把语言模型变得更会说）
   - 预训练（`train_pretrain.py`）：纯因果语言建模 CE loss（并可能叠加 MoE 的路由辅助损失 `aux_loss`）
   - 全量 SFT（`train_full_sft.py`）：把对话数据编码成输入 token + 只在回答片段计算 loss 的 mask
   - LoRA 微调（`train_lora.py`）：冻结主模型，仅训练通过 `model_lora.py` 注入到线性层上的 LoRA 分支
   - DPO（`train_dpo.py`）：用 preferred/rejected 两路计算偏好对齐损失
   - 蒸馏（`train_distillation.py`）：学生模型拟合教师模型分布（带温度的 KL / distillation loss），必要时叠加 CE
2. **强化学习链路**（把输出质量用奖励来拉）
   - PPO（`train_ppo.py`）
   - GRPO（`train_grpo.py`）
   - AgentRL（`train_agent.py`）：更复杂的 ToolCall 多轮交互与基于规则/奖励模型的评分
3. **采样/rollout 链路**（产生模型输出与 per-token logprob）
   - `trainer/rollout_engine.py`：提供统一接口，把“生成 + 计算每个生成 token 的 logprob”屏蔽成可插拔引擎（本地 PyTorch 或 HTTP + SGLang）
4. **推理/服务链路**（把训练好的权重变成可对话的服务）
   - `eval_llm.py`：本地交互与 `model.generate` 跑通
   - `scripts/chat_api.py`：Stream 输出（类似 OpenAI SSE 的风格）
   - `scripts/web_demo.py`：Streamlit 网页交互
   - `scripts/serve_openai_api.py`：FastAPI 提供 `/v1/chat/completions`
   - `scripts/eval_toolcall.py`：ToolCall 行为的评测（本地/OPENAI兼容 API）
   - `scripts/convert_model.py`：权重转换（torch -> transformers 格式，或 LoRA 合并等）

## 2. 关键目录与“你应该先读什么”

## 3. 全局张量维度“骨架”（全项目通用）

无论是预训练还是 RL，绝大多数代码遵循同一套张量骨架。

### 3.1 输入与输出的基本形状约定

对任意 Transformer 解码器模型，通常有：

- `input_ids`: `[B, T]`（batch 大小 B，序列长度 T）
- `attention_mask`: `[B, T]`（1 表示有效 token，0 表示 padding；有的地方可能为 None）
- 模型隐藏状态：`hidden_states`: `[B, T, C]`（C=hidden_size）
- 输出 logits：`logits`: `[B, T, V]`（V=vocab_size）
- 语言建模训练时，常见 teacher forcing 对齐是：
  - `shift_logits = logits[:, :-1, :]`，`shift_labels = labels[:, 1:]`
  - 也就是：预测 token `t` 的 logits 来自输入 token `t-1` 的位置

### 3.2 Attention 内部维度（抽象，不在本章展开到实现细节）

MiniMind 的实现使用：

- `q_proj/k_proj/v_proj` 产生 q/k/v 之后会 reshape 为：
  - `q`: `[B, T, n_heads, head_dim]`
  - `k/v`: `[B, T, n_kv_heads, head_dim]`（支持 GQA：kv head 数可能小于 q head 数）
- RoPE 作用在 q/k 上后，Attention 计算前会 transposition 与扩展：
  - 典型 Attention 计算会整理成：`[B, n_heads, T, head_dim]`
  - 当 kv head 数 < q head 数，会通过 `repeat_kv` 把 k/v 扩展到匹配的 `n_heads`

这些细节会在 `01` 中逐行落地到 `view/transpose/repeat_kv` 的具体写法与每一步张量维度变化。

## 4. 从 Dataset 到 Loss：训练链路中的数据流

以“全量 SFT”为例（`train_full_sft.py` + `SFTDataset`）说明端到端流程。

### 4.1 Dataset：把对话变成 token 与 labels

- `SFTDataset.__getitem__`：
  - 用 `tokenizer.apply_chat_template(...)` 把多轮 messages 拼成一个字符串 prompt
  - `tokenizer(prompt).input_ids[:max_length]` 得到长度为 `T=max_length` 的 token 序列
  - `labels = [-100 ...]` 的大框架会由 `generate_labels()` 决定：
    - 只对“assistant 输出区间”填入真实 token id
    - padding 对应的 label 会保持 `-100`，从而在 CE loss 中被 `ignore_index=-100` 掉

因此训练时：

- `input_ids`: `[B, T]`（long）
- `labels`: `[B, T]`（long，`-100` 用于忽略）

### 4.2 模型 forward：把 tokens 变成 logits

- `MiniMindForCausalLM.forward`：
  - 内部调用 `MiniMindModel.forward` 得到：
    - `hidden_states`: `[B, T, C]`
    - `aux_loss`: 标量（MoE 路由辅助项；dense 时通常为 0）
  - `lm_head` 把 hidden_states 投影到词表：
    - `logits`: `[B, T, V]`
  - 当 `labels` 不是 None 时：
    - 使用 `x = logits[..., :-1, :]` 与 `labels[..., 1:]` 对齐
    - `loss = cross_entropy(..., ignore_index=-100)`

所以最终训练 loss：

- `loss_total = res.loss + res.aux_loss`

这也在训练脚本里反复出现（预训练/全量 SFT/LoRA/DPO/蒸馏都遵循类似结构）。

## 5. RL 训练的“统一视图”：rollout + per-token logprob + mask

RL（PPO/GRPO/AgentRL）共同依赖两个关键张量：

1. **old logprob（旧策略下每个生成 token 的 logprob）**
2. **ref logprob（参考模型下同一 token 的 logprob）**

其中 rollout 引擎负责产出 old logprob：

- `rollout_engine.rollout(...)`：
  - 返回 `output_ids`: `[B*num_gen, P+R]`
  - 返回 `completion_ids`: `[B*num_gen, R]`
  - 返回 `per_token_logps`: `[B*num_gen, R]`（或返回后续被截切/对齐到与 completion mask 一致）

RL loss 里会用 mask 控制只在“completion 的有效 token”上计算 policy/value。

## 6. 检查点与分布式：训练脚本为何看起来都很像

几乎所有 `train_*.py` 都有相同的工程结构：

1. `init_distributed_mode()` 判定是否 DDP
2. `setup_seed()` 固定随机性
3. `lm_checkpoint()`：
   - 保存：`model/optimizer/scaler/scheduler/...` 等到 `../checkpoints/`
   - 加载：读取 resume pth 并校准 step（尤其考虑世界大小变化）
4. 数据 loader：
   - 非分布式：直接 DataLoader（或手工构造索引）
   - 分布式：使用 `DistributedSampler` + `SkipBatchSampler`
5. 训练循环：
   - 学习率 `get_lr(...)` 更新（余弦策略）
   - AMP autocast 与 GradScaler（float16 时启用）
   - 定期保存权重到 `save_dir` 与 `lm_checkpoint`

在 `04` 会把这些基础设施“逐行对齐”到你能看懂每个调用参数的意义。

## 8. 本章小结

你可以把 `minimind` 视作：

- `model_minimind.py` 定义 Transformer（`[B,T,C] -> logits [B,T,V]`）
- `lm_dataset.py` 把训练目标编码成 `labels/mask`
- `train_*.py` 用统一方式跑训练循环，叠加 `res.aux_loss`
- RL 系列用 `rollout_engine.py` 把“生成结果 + per-token logprob”送进 PPO/GRPO/AgentRL loss
- `scripts/*` 与 `eval_llm.py` 负责把 `generate` 或流式输出包装成服务/交互
